DNS缓存投毒

DNS缓存投毒又称DNS欺骗,DNS污染，是一种通过查找并利用DNS系统中存在的漏洞，将流量从合法服务器引导至虚假服务器上的攻击方式。

常见的三种DNS缓存投毒攻击:

1.盲目响应伪造

2.Kaminsky攻击

3.分片整理攻击

复习一下DNS相关内容(想看的可以点开看看)

漏洞复现: Kaminsky攻击

原文链接: https://seedsecuritylabs.org/Labs_16.04/PDF/DNS_Remote_new.pdf

ubuntu16下载: https://blog.csdn.net/dongjuexk/article/details/109334673

1.DNS缓存中毒攻击原理

原文:The goal of the attack is to launch the DNS cache poisoning attack on the local DNS server, such that when the user runs the dig command to find out www.example.com’s IP address, the local DNS server will end up going to the attacker’s nameserver ns.attacker32.com to get the IP address, so the IP address returned can be any number that is decided by the attacker. As results, the user will be led to the attacker’s web site, instead of to the authentic www.example.com
我的理解: DNS缓存中毒攻击的目标是本地DNS服务器,当正常用户尝试寻找www.example.com的ip地址时,本地DNS服务器就会向攻击者控制的DNS服务器发起查询请求.(在中毒前,应该是向权威DNS服务器进行查询请求)
正是因为本地DNS服务器向攻击者控制的DNS服务器发起了查询请求,因此我们可以随意构造虚假的返回包使本地DNS服务器信任并缓存,最终引导用户到我们的伪造网站.
#举个例子吧.(为了方便了解原理,这里的用户在本地没有缓存过任何DNS解析信息)
正常来说,用户想知道baidu.com的ip,需要让本地DNS服务器去查询baidu.com的权威域名服务器,baidu.com的权威域名服务器返回一个baidu.com的真实ip(一般使CDN服务的ip)

第一步,靶机1使其成为DNS服务器

1.安装bind9

sudo apt install bind9

2.配置DNS

vi /etc/bind/named.conf.options
#对这个文件进行四个操作
1.关闭validation
2.禁用dnssec(dnssec是Kaminsky漏洞的一个解决方案)
3.将缓存文件放在/var/cache/bind/dump.db(有缓存文件才能进行dns污染)
4.指定端口 33333(指定一个端口,否则端口随机化)

//DNSSEC-VALIDATION AUTO;

DNSSEC-ENABLE NO;

DUMP-FILE “/VAR/CACHE/BIND/DUMP.DB”;

QUERY-SOURCE PORT 33333;

LISTEN-ON-V6 { ANY; };

vi /etc/bind/named.conf
#对这个文件进行配置,可以让攻击机成为attacker32.com的权威域名服务器,这样当接收到attacker32.com的请求时,就不会尝试去寻找attacker32.com的权威服务器
#如果不配置,就会导致attacker32.com的权威服务器找不到,进而无法进行投毒
#在真实环境中,如baidu.com，不用当心这个问题，即使kakakak.baidu.com不存在,也会去baidu.com的权威域名服务器进行查询
#原文: Add the following zone entry to the /etc/bind/named.conf file. This entry indicates that for all queries of the attacker32.com domain, forward the queries to 10.0.2.8. This is equivalent to using 10.0.2.8 as the nameserver for the attacker32.com domain. Therefore, with this entry, the local DNS server will not try to find the IP address of attacker32.com’s nameserver; it already has the IP address. Please do not forget to include all those semicolons, or the configuration will be invalid.
